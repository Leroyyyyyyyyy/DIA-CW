# PMXT Backtesting Workflow

This document describes the PMXT/Nautilus-style replay workflow used as
supporting execution evidence for the coursework evaluation. It is research
documentation, not live trading functionality.

The final coursework tables are generated by the `poly-ok-check` evaluation
pipeline. PMXT replay artefacts can be imported into that pipeline when they are
available, but table values should still be regenerated through
`research/run/run_cw_experiment.py`.

## Workflow Summary

```text
1. Search for an event or market.
2. Expand the event and choose a market_slug.
3. Check the Yes/No token_index values.
4. Scan for PMXT windows with available quote data.
5. Write backtests/private/replays.json.
6. Run the replay runner.
7. Load historical PMXT order-book data.
8. Load the strategy and parameters.
9. Apply simulated fill and latency settings.
10. Run the backtest.
11. Inspect terminal output and any generated HTML report.
```

## Key Files

All paths below are relative to `vendor/prediction-market-backtesting/`.

```text
scripts/polymarket_find_markets.py
```

Searches markets, checks token IDs, scans PMXT availability, and can write
`replays.json`.

```text
backtests/private/replays.json
```

Tells the replay runner which market and time window to run.

```text
backtests/private/weather_backtest.py
```

Assembles data, strategy, fill/latency settings, reports, and starts the
backtest runner.

```text
strategies/ema_crossover.py
```

Contains the EMA strategy used by the smoke workflow.

```text
output/
```

Stores locally generated HTML reports and other output artefacts. This directory
is ignored and should not be treated as source data unless a specific result is
being imported into the evaluation config.

## Search for Events and Markets

Search active events:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --query hurricane `
  --active `
  --limit 10
```

Important fields in the output:

```text
event_slug
markets
title
```

Expand one event:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --event where-will-2026-rank-among-the-hottest-years-on-record
```

Important fields:

```text
market_slug
question
volume
active / closed
```

The replay uses `market_slug`, not `event_slug`.

## Confirm Yes/No Token Index

Replace the example market slug with the market being tested:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --market will-ludvig-aberg-win-the-2026-masters-tournament `
  --tokens
```

The output is usually:

```text
token_index outcome price token_id
0           Yes     ...
1           No      ...
```

Still check every market. Do not assume token index ordering without verifying
the output.

## Scan PMXT Availability

When the relevant time window is not known, scan automatically:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --market will-ludvig-aberg-win-the-2026-masters-tournament `
  --tokens `
  --scan-pmxt `
  --token-index 0
```

Default scan rules:

```text
closed market: scan backwards from endDate, with a default 7-day lookback
open market: scan backwards from now minus 3 hours, with a default 7-day lookback
if startDate is available, the scan is clipped to start after startDate
```

Default parameters:

```text
--scan-lookback-days 7
--scan-delay-hours 3
--scan-step-hours 6
--window-minutes 10
--max-windows 28
```

Useful output fields:

```text
PMXT scan range for ...
window_start
window_end
quotes
```

Only windows with `quotes > 0` should be used for replay. Use `--scan-all` to
print empty windows too.

Examples:

```powershell
# Try a 10-minute window every hour.
--scan-step-hours 1 --window-minutes 10

# Scan a longer period.
--scan-lookback-days 14 --max-windows 100
```

An event-level scan is also possible, but it can be slow when the event contains
many markets:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --event where-will-2026-rank-among-the-hottest-years-on-record `
  --scan-pmxt `
  --token-index 0 `
  --max-windows 6
```

## Write `replays.json`

The recommended path is to let the scan script write the replay config:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --market will-ludvig-aberg-win-the-2026-masters-tournament `
  --tokens `
  --scan-pmxt `
  --token-index 0 `
  --write-replays backtests\private\replays.json
```

Example output:

```json
[
  {
    "market_slug": "will-ludvig-aberg-win-the-2026-masters-tournament",
    "token_index": 0,
    "start_time": "2026-04-05T00:00:00Z",
    "end_time": "2026-04-05T00:10:00Z",
    "quotes": 346,
    "records": 692,
    "order_book_deltas": 346
  }
]
```

The script writes only windows with `quotes > 0`. If no valid windows are found,
it writes an empty list and the runner should reject the replay file.

For a known UTC window:

```powershell
.venv\Scripts\python.exe scripts\polymarket_find_markets.py `
  --market will-ludvig-aberg-win-the-2026-masters-tournament `
  --tokens `
  --check-pmxt `
  --token-index 0 `
  --start-time 2026-04-05T05:10:00Z `
  --end-time 2026-04-05T05:20:00Z
```

Use `--scan-start` and `--scan-end` only when the scan range needs to be forced:

```powershell
--scan-start 2026-04-05T00:00:00Z --scan-end 2026-04-06T00:00:00Z
```

## Data Source Configuration

The PMXT data-source order used by the runner is:

```python
DATA = MarketDataConfig(
    platform=Polymarket,
    data_type=QuoteTick,
    vendor=PMXT,
    sources=(
        "local:data/pmxt_raws",
        "archive:r2v2.pmxt.dev",
        "relay:209-209-10-83.sslip.io",
    ),
)
```

Meaning:

```text
local: read local raw data first
archive: fetch from the PMXT archive if local data is unavailable
relay: fallback when the archive is unavailable
```

## Replay Configuration

`backtests/private/weather_backtest.py` first reads:

```text
backtests/private/replays.json
```

The runner then builds replay entries equivalent to:

```python
REPLAYS = (
    QuoteReplay(
        market_slug="will-ludvig-aberg-win-the-2026-masters-tournament",
        token_index=0,
        start_time="2026-04-05T00:00:00Z",
        end_time="2026-04-05T00:10:00Z",
    ),
)
```

If `replays.json` does not exist, the runner falls back to an internal smoke
window.

## Smoke Strategy Parameters

The current smoke runner uses short EMA periods so the replay can confirm that
the execution path produces fills:

```python
"fast_period": 8,
"slow_period": 21,
"entry_buffer": 0.0,
"take_profit": 0.01,
"stop_loss": 0.01,
```

These are smoke parameters, not optimised strategy results.

## Filtering and HTML Reports

For debugging, the runner can lower filters and emit HTML:

```python
EXPERIMENT = build_replay_experiment(
    ...
    min_quotes=1,
    min_price_range=0.0,
    emit_html=True,
    chart_output_path="output",
)
```

HTML generation is a capability of the vendored backtesting project and is
triggered through `emit_html=True`.

If a very short replay window has only one fill, the legacy chart may warn about
`x_range=None`. The backtest result can still print successfully. Wider replay
windows or more active market windows usually produce more stable charts.

## Run the Backtest

```powershell
.venv\Scripts\python.exe backtests\private\weather_backtest.py
```

One previously verified smoke replay produced:

```text
Market                                                Quotes  Fills   PnL (USDC)
will-ludvig-aberg-win-the-2026-masters-tournament        346      1      +0.0000
TOTAL                                                             1      +0.0000
```

A zero PnL smoke result does not necessarily mean the replay failed. In this
case it means the automatically discovered 10-minute window produced only one
fill and did not create a realised profit/loss path. To inspect stronger PnL
movement, expand the window or scan for more active windows.
